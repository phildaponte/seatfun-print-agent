## 2026-05-20

- **docs (architecture + distribution)**: Made the **\"Option A\" loopback-only constraint explicit** — agent binds to `127.0.0.1` (never `0.0.0.0`), so only the device running the agent can submit print jobs. Phones / tablets / other laptops on the same Wi-Fi cannot print, by design (a LAN-bound agent would let any device on the venue Wi-Fi submit prints). This is documented in `docs/architecture.md → Security posture` with a forward link to a possible v1.1 \"LAN mode\". Also affirmed **Windows and macOS as co-priority for v1** — most of our client's organizers run Windows, so a Mac-only v1 is not a v1; both installers must ship together. Updated `docs/architecture.md → Cross-platform strategy` with concrete platform-shim plan (Tauri `keyring` / `autostart` / `tray` plugins, OS-specific log paths). Corrected cost figures in `docs/distribution.md`: Apple Developer membership is already paid (so $0 incremental for Mac); the only variable cost is the optional Windows code-signing cert ($0 unsigned + SmartScreen click-through, $30/yr Certum, $120/yr Azure Trusted Signing, $300+ for EV). Realistic v1 total: **$0–$30/year**, not the $300–500 originally quoted. Why: the user identified that multi-device printing was unclear in the original docs, and that v1 cost concerns were inflated. Both materially affect the build plan. Docs: [docs/architecture.md](./docs/architecture.md), [docs/distribution.md](./docs/distribution.md).\n\n- **docs (Tailscale testing guide)**: Added `docs/testing-with-tailscale.md` — step-by-step runbook for testing the v0 CLI agent against a remote BOCA Lemur printer on a client's LAN via Tailscale subnet routing. Covers Tailscale install, user-invite flow, subnet routing setup on the client's Mac (`tailscale up --advertise-routes`), printer IP discovery via BOCA self-test, connectivity verification with `nc`, `.env` setup, the iteration loop with `npm run smoke`, a first-print validation checklist (layout, QR scan, accented chars, tear line), and a troubleshooting section. Why: the v0 dev loop is meaningless without remote LAN access — the developer's Mac and the client's printer are on different networks. Docs: [docs/testing-with-tailscale.md](./docs/testing-with-tailscale.md).

- **chore (repo init)**: Initialized the repo with the four planning docs imported from `seatfun-dashboard/docs/seatfun-print-agent/`: `README.md`, `docs/architecture.md`, `docs/distribution.md`, `docs/protocol.md`. Added documentation rule under `.windsurf/rules/` (mirrors the dashboard's). Added frontmatter to all four docs. Why: production discipline — every doc has owner + status + last change, every code change ships with doc + CHANGELOG updates.

- **feat (v0 FGL renderer)**: Added `src/fgl/` — pure renderer producing FGL command streams for the BOCA Lemur-S (FGL46, 2"×5.5" tear stock). Modules: `commands.ts` (typed primitives `<RC>`, `<F>`, `<QR>`, `<LOGO>`, `<p>`, `<CAN>`), `types.ts` (`TicketRenderData` matching `tickets[].fields` from [protocol.md](./docs/protocol.md)), `template.ts` (`renderTicket(data)` implementing the layout from `seatfun-dashboard/docs/PRD — Printing tickets system…`), `index.ts` (public exports). Golden test fixtures in `src/fgl/__golden__/` (`basic.json` + `basic.fgl`). Tests in `src/fgl/template.test.ts` (vitest). The renderer is pure: same input → same bytes; no I/O, no clocks, no randomness. This is what enables byte-exact golden tests and lets the dashboard preview tickets identically to how the printer renders. Logo handling is deferred: v0 emits text only; v1 will upload `logo-black.png` to printer flash via FGL `<SL>` and reference via `<LOGO1>`. Why: the renderer is the most platform-independent piece — getting it right with golden tests before any I/O code is the safest place to start. Docs: [docs/architecture.md → `src/fgl/`](./docs/architecture.md).

- **feat (v0 CLI agent)**: Added `src/server/`, `src/printer/`, `src/index.ts` — the v0 CLI agent. HTTP server bound to `127.0.0.1:9787` (loopback only, never `0.0.0.0`), implements `GET /v1/health` (no auth) and `POST /v1/print` (bearer-token auth via `SEATFUN_AGENT_TOKEN` env). Per-ticket flow: render FGL → open TCP socket to `PRINTER_IP:9100` → write bytes → close → next ticket (50ms gap to avoid printer buffer overflow). Returns the per-ticket `results[]` shape from `docs/protocol.md`. Errors mapped to `printer_unreachable` / `render_error` per the protocol. Token never logged. Why: enough surface to run the first end-to-end print spike with the dashboard the moment the printer IP is in hand. v0 intentionally skips: pairing handshake, OS-keychain storage, queue, heartbeat, `/v1/cancel`, `/v1/status`, `/v1/test-print`. Those land in v1. Docs: [docs/architecture.md](./docs/architecture.md), [docs/protocol.md](./docs/protocol.md). Code: `src/server/http.ts`, `src/server/routes/print.ts`, `src/server/routes/health.ts`, `src/server/middleware/auth.ts`, `src/printer/client.ts`, `src/index.ts`.

- **chore (tooling)**: Added npm + TypeScript + vitest setup. Files: `package.json`, `tsconfig.json`, `.gitignore`, `.editorconfig`, `.env.example`, `fixtures/sample-job.json`. Scripts: `dev` (tsx watch), `build` (tsc), `start`, `test` (vitest), `lint` (tsc --noEmit), `smoke` (real-printer smoke against `PRINTER_IP`). Node 20+ required (uses built-in `fetch`, `node:test`-compatible APIs, `--enable-source-maps`). Why: minimum tooling to run the FGL golden tests and boot the agent locally.
